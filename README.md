# Basic MMO RPG

Стартовый 2D online RPG / UO-like prototype на Python.

Текущий этап: MVP 10, где несколько клиентов подключаются к authoritative websocket-серверу, входят по имени персонажа, видят друг друга, общаются в чате, восстанавливают сохраненную позицию после reconnect-а, взаимодействуют с NPC, торговцем, crafting-станциями и lootable-объектами, хранят инвентарь, экипируют предметы в слоты `main_hand` и `chest`, проходят первые gathering- и crafting-петли, взаимодействуют с живой овцой Барбарой в загоне, атакуют тренировочный манекен и сражаются с первым non-aggro противником.

## Быстрый запуск

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -e ".[dev]"
```

В первом терминале запусти сервер:

```powershell
python -m basic_mmo_rpg.server.app
```

Во втором и третьем терминалах запусти клиентов:

```powershell
python -m basic_mmo_rpg.client.app --server ws://127.0.0.1:8765 --name Alice
python -m basic_mmo_rpg.client.app --server ws://127.0.0.1:8765 --name Bob
```

## Игровые скиллы

Сервер хранит persistent-скиллы персонажа в SQLite и отправляет клиенту `skills_updated`. Окно скиллов открывается и закрывается клавишей `K`. Сейчас скиллы влияют на gathering: Mining для `rock` и `cave wall`, Lumberjacking для деревьев и Fishing для водных тайлов; Tailoring влияет на крафт ткани, Healing - на применение бинтов. При росте скилла игрок получает feed-сообщение, которое также попадает в журнал.

Актуальные формулы, стартовые значения и demo-cap описаны в [CHARACTER_SKILLS.md](CHARACTER_SKILLS.md). Ниже README кратко описывает пользовательский flow gathering-а без дублирования всех формул.

Управление движением: удерживать `ПКМ`, направление выбирается по одному из 8 секторов курсора относительно персонажа. `Enter` открывает ввод чата, повторный `Enter` отправляет сообщение, `Esc` отменяет активный ввод или закрывает открытое NPC/vendor-окно. `J` показывает/скрывает журнал последних сообщений текущей сессии. `B` показывает/скрывает инвентарь и paperdoll. `K` показывает/скрывает скиллы. `F1` показывает/скрывает легенду горячих клавиш. В панели `B` клик по экипируемому предмету надевает его в подходящий слот, клик по занятым слотам `В руке` или `Грудь` снимает предмет. `Tab` включает/выключает боевой режим; при выключении серверный auto-attack сбрасывается. `H` отправляет server-authoritative запрос применения бинта. Наведение курсора на NPC, овцу или lootable-объект показывает имя объекта, наведение на воду, дерево, камень или стену пещеры подсвечивает ресурсный тайл, `F` отправляет запрос взаимодействия с объектом или ресурсным тайлом строго под курсором. В боевом режиме attackable-объект под курсором подсвечивается красным, клик по нему выбирает цель auto-attack, а выбранная цель отмечается темно-красным перекрестием. Клик по другому персонажу показывает его никнейм на 3 секунды. При смерти открывается окно `Вы погибли` с кнопкой `Возродиться`. Закрытие клиента сейчас только через крестик окна. Остановка сервера: `Ctrl+C`.

Ручные команды вводятся в чат и обрабатываются сервером только для текущего персонажа. Сейчас доступна `/setspeed <0-600>`: например, `/setspeed 220` меняет runtime-скорость персонажа до перезахода или рестарта сервера и не рассылается как обычное сообщение чата.

На стартовой карте внутри помещения стоят NPC `Funday`, `Jack Lumber`, `Kopai`, торговец `Bjorn`, `Горн`, `Наковальня` и `Прядильный станок`. Если персонаж находится не дальше 64 пикселей и курсор наведен на NPC или интерактивный объект, клавиша `F` открывает server-authoritative окно с репликой и доступными опциями. Если нужного tutorial-инструмента еще нет, в окне NPC появляется опция его получить; сервер добавляет предмет в persistent-инвентарь и отправляет клиенту `inventory_updated`. NPC не экипируют инструмент автоматически: игрок сам берет его в руку через панель `B`. Обмены у NPC оформлены как repeatable-квесты: открытая, но невыполнимая опция остается серой и показывает прогресс предметов, а успешная сдача атомарно списывает предметы, выдает награду, записывает факт сдачи и оставляет NPC-окно открытым с обновленным состоянием. У Bjorn есть отдельная опция `Торговля`, открывающая vendor-окно. Пока он продает только `Железная кираса` за `Gold x25`; продажа предметов игроком еще не реализована. `Горн` пытается переплавить `Железная руда x1` в `Железный слиток x1` с шансом успеха `0.8`; при неудаче руда тратится, но слиток не появляется. `Наковальня` атомарно меняет `Железный слиток x35` на `Железная кираса x1`. `Прядильный станок` делает `Ткань x4` из `Шерсть x1` с шансом от Tailoring; при неудаче шерсть тратится. Там же можно без fail сделать `Бинт x1` из `Ткань x1`.

Рыбалка работает через водные тайлы: если курсор наведен на воду рядом с персонажем, клавиша `F` отправляет `interact_requested` с координатами тайла. Без удочки в руке сообщение `Нужна удочка в руке` появляется в нижнем логе событий без записи в журнал. С экипированной удочкой персонаж входит в состояние подготовки: длительность зависит от `Fishing`, и в это время он не двигается, не атакует, не взаимодействует и не меняет предмет в руке. После подготовки сервер проверяет успех по `Fishing`; при успехе персонаж получает `Рыба`, при неудаче видит `Рыба сорвалась` в нижнем логе событий. Эти результаты попадают в журнал, а завершенная валидная попытка может повысить `Fishing`. В NPC-окне `Funday` repeatable-опция обменивает `Рыба x2` на `Gold x1`.

Рубка работает через тайлы деревьев: без `Топор для рубки` в руке сообщение `Нужен топор в руке` появляется в нижнем логе событий без записи в журнал. С экипированным топором после общей подготовки сервер проверяет успех по `Lumberjacking`: успешная рубка дает `Бревно`, а при достаточном значении скилла может дать два бревна за одну попытку. Результаты `Вы нарубили древесины` или `Не удалось нарубить древесины` попадают в журнал, а завершенная валидная попытка может повысить `Lumberjacking`. В NPC-окне `Jack Lumber` repeatable-опция обменивает `Бревно x5` на `Gold x1`.

Добыча камня работает через тайлы `rock` и `cave wall`: без `Кирка` в руке сообщение `Нужна кирка в руке` появляется в нижнем логе событий без записи в журнал. С экипированной киркой после общей подготовки сервер сначала проверяет успех по `Mining`, а затем при успехе выбирает добытый ресурс: на низких значениях доступен только `Камень`, позднее появляется шанс добыть `Железная руда`. Результаты добычи или неудачи попадают в журнал, а завершенная валидная попытка может повысить `Mining`. В NPC-окне `Kopai` repeatable-опция обменивает `Камень x3` на `Gold x1`.

В правой верхней части карты есть загон с калиткой и овцой `Овца`; Fogu называет ее Барбарой в своей реплике. Калитка переключается клавишей `F` под курсором: закрытая калитка является solid-сущностью, открытая становится проходимой для игрока и овцы. Овца имеет `15 HP`, хранит состояние шерсти и каждые 2 секунды выбирает направление: если соседний тайл свободен, она плавно идет туда 2 секунды, иначе стоит на месте 2 секунды; затем ждет еще 2 секунды до следующей попытки. В NPC-окне `Fogu` можно получить `Ножницы`, если их еще нет. С ножницами в руке можно состричь с овцы `Шерсть`; через 30 секунд сервер снова помечает шерсть как выросшую. В NPC-окне `Fogu` repeatable-опция обменивает `Шерсть x1` на `Gold x1`.

В нижней правой части карты стоит `Тренировочный манекен`. Это component-based world-object с независимыми `lootable`, `combat` и `respawn` компонентами: при первом взаимодействии через `F` сервер persistent-записывает loot-claim персонажа, добавляет в инвентарь `Ржавый меч` и отправляет результат `Вы вытащили Ржавый меч из манекена`, который попадает в журнал. Повторные взаимодействия этого же персонажа с манекеном ничего не делают. Loot доступен независимо от того, целый манекен или разрушенный. `Ржавый меч` можно взять в руку через панель `B`.

Базовая PvE-боевка работает через `Tab` и клик по attackable-объекту. Сервер запоминает цель и выполняет auto-attack authoritative: если персонаж вне дистанции 64 px, атака молчит и ждет возвращения в радиус; если цель в радиусе, удар проходит по cooldown. Без подходящего оружия используется unarmed-урон `1-2`, `Ржавый меч` бьет на `3-6` с cooldown `1.2s`, шанс попадания пока фиксированный `85%`. `Железная кираса` надевается в слот `chest`, дает `Броня +2` и уменьшает входящий урон на 2, но итоговый урон не может стать ниже 1. Каждый hit/miss отправляется как `combat_event` и попадает в журнал: `Вы атаковали Тренировочный манекен: -4` или `Вы промахнулись`. При входящем ударе броня дописывает поглощение в журнал, например `Кабан атаковал вас: -1 (1/2 поглощено броней)`. При `0 HP` манекен становится `Разрушенный тренировочный манекен`, остается solid, auto-attack сбрасывается, а через 60 секунд server-time восстанавливается.

Бинты применяются клавишей `H`. Если бинтов нет или персонаж не ранен, сервер отправляет сообщение в нижний лог и журнал без списания предмета. При валидном применении сервер сразу списывает `Бинт x1`, сбрасывает текущий отсчет attack swing, но не сбрасывает выбранную цель и не блокирует движение; цель auto-attack можно сменить во время перевязки. После задержки, зависящей от Healing, сервер делает roll успеха, лечит HP при успехе, отправляет feed/journal-сообщение вида `Alice: вылечил +3 бинтом` и запускает новый полный swing cooldown.

В левом нижнем углу карты гуляет `Кабан`. Пока его не атаковали, он ведет себя как neutral creature; после любой атаки вступает в бой, преследует атакующего, но срывает погоню при слишком большом удалении от дома. Кабан атакует на дистанции 64 px, имеет собственные hit chance, урон и cooldown, а цель выбирает по наибольшему нанесенному ему урону. При смерти кабан скрывается, через 60 секунд respawn-ится дома с 50% HP, а на земле остается проходимый и не attackable труп на 5 минут. Первый персонаж, взаимодействующий с трупом, получает `Кожа x2`. У игрока есть `30 HP`; при смерти он становится невидимым, не может двигаться, атаковать или взаимодействовать, видит `Вы погибли` и возрождается кнопкой у крестa на карте с `15 HP`.

По умолчанию сервер хранит позиции персонажей в `data/game.sqlite3`. Локальные SQLite-файлы игнорируются git-ом.


## Редактор карты

Редактор карты живет отдельно от игрового клиента и сервера:

```powershell
.\.venv\Scripts\python.exe scripts\run_map_editor.py --map assets\maps\starter_map.json
```

Управление:

- `F1` - показать/скрыть окно hotkey'ев;
- `1..9` - выбрать тайл из палитры;
- ЛКМ по карте - рисовать выбранным тайлом;
- удержание ЛКМ - рисовать кистью 1x1;
- ПКМ по карте - выбрать тайл под курсором;
- ЛКМ по entity - выбрать и перетащить entity;
- `Ctrl` при перетаскивании entity - притянуть центр entity к центру тайла;
- `Ctrl+D` - скопировать выбранную creature-entity; NPC не копируются;
- колесо мыши - вертикальная прокрутка карты;
- `Shift` + колесо мыши - горизонтальная прокрутка карты;
- стрелки или `WASD` - прокрутка карты;
- средняя кнопка мыши - перетащить видимую область карты;
- `Z` - переключить масштаб `1x` / `0.5x`;
- `Ctrl+S` - сохранить карту.

Перед первой записью за сессию редактор создает backup рядом с картой:
`starter_map.json.bak`, `starter_map.json.bak.1` и так далее. Backup-файлы игнорируются git-ом.
Сохранение проходит через загрузчик карты, поэтому карта с entity на solid-тайле или за границей мира не будет записана.

Новая карта произвольного размера создается командой:

```powershell
.\.venv\Scripts\python.exe scripts\run_map_editor.py --map assets\maps\my_map.json --new 60x35
```

Новая карта берет `legend` и entities из `starter_map.json`, заполняет все поле травой `.` и открывается в редакторе.
Внешняя стена не создается: границей мира остается размер карты. Все entities, кроме gate, ставятся в левом верхнем углу,
чтобы их можно было потом перетащить на нужные места. Если файл уже существует, редактор не перезапишет его без флага
`--overwrite`.

## Структура

```text
src/basic_mmo_rpg/
  client/   pygame-ce: ввод, камера, рендер, UI, websocket-клиент
  domain/   чистая игровая логика: карта, геометрия, сущности мира, инвентарь, движение, коллизии
  server/   asyncio/websockets authoritative-сервер и серверное состояние мира
  shared/   JSON-протокол сообщений между клиентом и сервером
  storage/  загрузка JSON-карты и сохранение персонажей в SQLite
assets/maps/
  starter_map.json
scripts/
  run_client.py
  run_server.py
tests/
```

## Сетевая модель

Клиент сначала отправляет имя персонажа, а дальше отправляет намерения движения, сообщения чата и запросы взаимодействия:

```text
client -> server: join_requested
client -> server: move_requested
client -> server: chat_sent
client -> server: interact_requested
client -> server: interaction_option_selected
client -> server: equip_item_requested
client -> server: unequip_item_requested
client -> server: vendor_buy_requested
client -> server: attack_requested
client -> server: stop_attack_requested
client -> server: respawn_requested
```

Сервер хранит активные сессии, кикает старое подключение при повторном входе тем же именем, применяет движение через доменную функцию `move_player`, проверяет коллизии с тайлами и runtime-объектами, двигает овцу и neutral/enemy creature-сущности, проверяет дистанцию взаимодействия с NPC, калиткой, creature-сущностями, lootable-объектами, crafting-станциями и ресурсными тайлами, собирает окна взаимодействия и заново валидирует выбранные опции, выполняет правила инвентаря, repeatable-обменов, crafting-обменов, одноразового loot-а и PvE-боевки, сохраняет позиции, инвентарь, экипировку, persistent loot-claims и счетчики repeatable-квестов в SQLite и рассылает snapshot мира:

```text
server -> client: connection_accepted
server -> client: world_snapshot
server -> client: chat_message
server -> client: interaction_result
server -> client: interaction_menu_opened
server -> client: vendor_opened
server -> client: inventory_updated
server -> client: equipment_updated
server -> client: combat_event
server -> client: entity_removed
```

Клиент не решает, где реально находится персонаж в online-режиме, что экипировано и была ли успешной добыча; он рисует состояние, полученное от сервера.
