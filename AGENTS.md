# Agent Project Context

## Проект

Мы разрабатываем небольшой 2D online RPG / UO-like prototype на Python.

Это не попытка сразу сделать полноценную MMORPG. Цель текущего этапа - играбельный прототип для 2-10 друзей с чистой архитектурой и быстрыми MVP-итерациями.

## Важное решение: локальный режим не поддерживаем

Локальную/offline-версию клиента больше не поддерживаем.

Клиент должен запускаться как multiplayer-client и подключаться к websocket-серверу. CLI-клиент требует `--name`; `--server` по умолчанию указывает на `ws://127.0.0.1:8765`.

Не добавлять новые gameplay-фичи только в локальный режим и не дублировать authoritative-логику на клиенте. Клиент может делать prediction/interpolation для UX, но источник истины - сервер.

## Стек

- Language: Python 3.12+
- Client: pygame-ce
- Server: Python asyncio
- Network: websockets over TCP
- Serialization v1: JSON
- Database v1: SQLite
- Map format сейчас: JSON, с перспективой Tiled `.tmx` или `.json`
- Tests: pytest
- Lint/format: ruff
- Type checking: mypy включен и должен проходить для `src`

## Архитектурные правила

Сервер authoritative:

- клиент отправляет намерения и запросы;
- сервер проверяет и применяет результат;
- клиент не решает, где реально находится персонаж;
- клиент не решает, успешно ли взаимодействие;
- клиент не решает, получил ли игрок предмет или прошел ли сквозь препятствие.

Примеры сообщений:

```text
client -> server: join_requested
client -> server: move_requested
client -> server: chat_sent
client -> server: interact_requested
client -> server: equip_item_requested
client -> server: unequip_item_requested
client -> server: attack_requested
client -> server: stop_attack_requested
client -> server: respawn_requested
client -> server: apply_bandage_requested

server -> client: connection_accepted
server -> client: world_snapshot
server -> client: chat_message
server -> client: interaction_result
server -> client: inventory_updated
server -> client: equipment_updated
server -> client: combat_event
server -> client: entity_removed
server -> client: error
```

Разделение модулей:

- `client/` - pygame, ввод, камера, рендер, UI, websocket-клиент;
- `server/` - websocket server, игровые сессии, авторизация, рассылка событий;
- `shared/` - протокол сообщений, DTO/schema, сериализация;
- `domain/` - чистая игровая логика без pygame/websocket;
- `storage/` - загрузка карты и сохранение persistent-состояния.

Сущности карты сейчас component-based. `kind` описывает широкую категорию/визуальный тип, а gameplay-возможности задаются компонентами вроде `interaction`, `lootable`, `combat`, `respawn`, `gate` и `shearable`. Не добавлять новые gameplay-возможности как новые `kind`, если они лучше выражаются компонентом.

## Правила реализации

- 2D only. 3D не рассматривается.
- Визуальный референс: Ultima Online, но без требования идеально повторять изометрию.
- Быстрый играбельный прототип важнее идеального движка.
- Новую игровую механику сначала проводить через server authoritative flow.
- Не расширять JSON-протокол ad hoc без helper-функций сериализации/валидации в `shared/protocol.py`.
- Если объект влияет на коллизии, учитывать его и на сервере, и в client-side prediction.
- Все NPC должны быть коллизионными объектами карты.
- SQLite-файлы, cache и runtime-артефакты не коммитить.
- Докстринги и поясняющие комментарии писать на русском.
- Имена модулей, классов, функций и переменных оставлять в обычном Python-стиле проекта.

## UX-правила текущего клиента

- `WASD` или стрелки - движение.
- `Enter` - начать ввод чата.
- Повторный `Enter` - отправить сообщение.
- `Esc` - отменить активный ввод чата.
- `J` - показать/скрыть журнал сообщений.
- `B` - показать/скрыть инвентарь и paperdoll.
- `K` - показать/скрыть окно скиллов.
- `H` - применить бинт.
- `F1` - показать/скрыть легенду горячих клавиш.
- `Tab` - включить/выключить боевой режим.
- При выключении боевого режима клиент отправляет `stop_attack_requested`.
- Клик по экипируемому предмету в панели `B` - запросить экипировку в `main_hand`.
- Клик по занятому слоту `В руке` в панели `B` - запросить снятие предмета.
- В боевом режиме наведение на attackable-объект подсвечивает его красным.
- В боевом режиме клик по attackable-объекту отправляет `attack_requested` и запоминает выбранную цель.
- Выбранная боевая цель отмечается темно-красным перекрестием.
- Над выбранной боевой целью показывается маленькая полоска HP.
- Клик по другому игроку - временно показать его никнейм.
- Наведение на NPC - показать имя NPC.
- Наведение на creature-сущность - показать имя объекта.
- Наведение на lootable-сущность - показать имя объекта.
- Наведение на водный тайл, дерево или камень - подсветить ресурсный тайл.
- `F` - отправить `interact_requested` для объекта или ресурсного тайла строго под курсором.
- Если персонаж погиб, клиент показывает окно `Вы погибли`; кнопка `Возродиться` отправляет `respawn_requested`.
- Мертвый персонаж невидим и не может двигаться, атаковать или взаимодействовать.
- Клиент закрывается через крестик окна, не через `Esc`.

## Правила текущей игровой петли MVP 10

- `Funday` говорит `Иди и поймай мне рыбу`.
- Если у персонажа нет `Удочка`, `Funday` выдает ее один раз через явное правило `add_item_if_absent`.
- Stack limit предмета остается инвариантом предмета, но не является источником tutorial-ограничения выдачи удочки.
- NPC не экипируют tutorial-инструмент автоматически: игрок сам берет его в руку через панель `B`.
- Экипированный предмет сейчас остается в инвентаре, а `main_hand` хранит ссылку на `item_id`; настоящий перенос предмета в слот paperdoll записан в TODO.
- Без удочки в руке взаимодействие с водой показывает над персонажем `Нужна удочка в руке` и не добавляет запись в журнал.
- Взаимодействие с неводным тайлом ничего не показывает.
- При успешной рыбалке персонаж получает `Рыба`, видит `Вы поймали рыбу`, сообщение попадает в журнал.
- При неудачной рыбалке персонаж видит `Рыба сорвалась`, сообщение попадает в журнал.
- Если у персонажа есть минимум две рыбы, взаимодействие с `Funday` атомарно списывает `Рыба x2` и выдает `Gold x1`.
- `Jack Lumber` говорит `Наруби немного древесины`.
- Если у персонажа нет `Топор для рубки`, `Jack Lumber` выдает его один раз через явное правило `add_item_if_absent`.
- Без топора в руке взаимодействие с деревом показывает над персонажем `Нужен топор в руке` и не добавляет запись в журнал.
- При успешной рубке персонаж получает `Бревно`, видит `Вы нарубили древесины`, сообщение попадает в журнал.
- Если у персонажа есть минимум пять бревен, взаимодействие с `Jack Lumber` атомарно списывает `Бревно x5` и выдает `Gold x1`.
- `Kopai` говорит `Накопай мне чего-нибудь`.
- Если у персонажа нет `Кирка`, `Kopai` выдает ее один раз через явное правило `add_item_if_absent`.
- Без кирки в руке взаимодействие с камнем показывает над персонажем `Нужна кирка в руке` и не добавляет запись в журнал.
- При успешной добыче персонаж получает `Камень`, видит `Вы добыли камень`, сообщение попадает в журнал.
- При неудачной добыче персонаж видит `Не удалось добыть камень`, сообщение попадает в журнал.
- Если у персонажа есть минимум три камня, взаимодействие с `Kopai` атомарно списывает `Камень x3` и выдает `Gold x1`.
- Cooldown gathering-а сейчас общий для рыбалки, рубки и добычи камня, чтобы переключение ресурса не обходило задержку.
- Загон и калитка являются частью starter-map.
- Состояние калитки runtime-only: после рестарта сервера она возвращается к состоянию из карты.
- `F` по калитке переключает состояние `open/closed`; закрытая калитка solid, открытая проходима.
- Если проход занят игроком или другой solid-сущностью, открытую калитку нельзя закрыть, игрок видит `Проход занят`.
- `Овца` является `creature`-сущностью с `15 HP`; Fogu называет ее Барбарой в реплике, атаки пока не реализованы.
- Овца каждые 2 секунды выбирает направление, 2 секунды плавно движется в свободный соседний тайл или стоит на месте при заблокированной попытке, затем ждет 2 секунды до следующего выбора.
- При проверке движения Барбары занятыми считаются solid tiles, solid entities и игроки.
- `Fogu` говорит `Постриги мою Барбару`.
- Если у персонажа нет `Ножницы`, `Fogu` выдает их один раз через `add_item_if_absent`.
- Без ножниц в руке взаимодействие с Барбарой показывает над персонажем `Нужны ножницы в руке` и не добавляет запись в журнал.
- Если шерсть еще не выросла, взаимодействие с Барбарой показывает `Шерсть еще не выросла` и не добавляет запись в журнал.
- С ножницами в руке персонаж стрижет овцу, получает `Шерсть`, видит `Вы состригли шерсть с овцы`, сообщение попадает в журнал.
- Через 30 секунд server-time шерсть Барбары снова вырастает.
- Если у персонажа есть минимум одна шерсть, взаимодействие с `Fogu` атомарно списывает `Шерсть x1` и выдает `Gold x1`.
- `Прядильный станок` является solid world-object на starter-map и открывает crafting-окно по `F`.
- Рецепт `Шерсть x1 -> Ткань x4` на прядильном станке зависит от Tailoring: до `15.0%` шанс успеха `0.5`, далее линейно растет до `1.0` при `40.0%`.
- При неудачном крафте ткани шерсть тратится, ткань не появляется, валидная попытка может повысить Tailoring.
- Рецепт `Ткань x1 -> Бинт x1` на прядильном станке проходит без fail и сейчас не качает Tailoring.
- В нижней правой части карты стоит `Тренировочный манекен` как первый `lootable` world-object.
- При первом взаимодействии с манекеном сервер persistent-записывает loot-claim персонажа и выдает `Ржавый меч`.
- Игрок видит `Вы вытащили Ржавый меч из манекена`, сообщение попадает в журнал.
- Повторные взаимодействия того же персонажа с манекеном ничего не делают.
- `Ржавый меч` можно взять в руку через панель `B`.
- Loot манекена независим от его разрушения: `F`-взаимодействие может выдать меч один раз, даже если манекен разрушен.
- Манекен является `attackable` и `destructible` объектом с `20 HP`.
- Auto-attack выбирается кликом по манекену в боевом режиме.
- Если персонаж дальше `64 px`, auto-attack не сбрасывается и молчит до возвращения в радиус.
- При выключении боевого режима auto-attack сбрасывается через `stop_attack_requested`.
- Шанс попадания игрока пока фиксированный: `0.85`.
- Без подходящего оружия используется unarmed-атака `1-2`.
- `Ржавый меч` наносит `3-6` урона с cooldown `1.2s`.
- Каждый удар или промах отправляется как `combat_event` и попадает в журнал: `Вы атаковали Тренировочный манекен: -4` или `Вы промахнулись`.
- При `0 HP` манекен становится `Разрушенный тренировочный манекен`, остается solid, выбранная цель сбрасывается.
- Через 10 секунд server-time манекен восстанавливается до `20 HP` и снова становится attackable.
- В левом нижнем углу карты находится non-aggro противник `Кабан`.
- Пока кабан не заагрен, он гуляет по тем же runtime-правилам, что и овца.
- После любой атаки игрока кабан вступает в бой и двигается к атакующему.
- Если кабан слишком далеко уходит от дома, leash сбрасывает aggro и возвращает его к точке spawn-а.
- Кабан выбирает боевую цель по наибольшему нанесенному ему урону.
- Кабан атакует игрока на дистанции `64 px` по своим combat-параметрам.
- У игрока `30 HP`, здоровье runtime-only.
- Если игрок умирает, сервер сбрасывает его auto-attack, клиент показывает `Вы погибли`, персонаж становится невидимым и не может двигаться, атаковать или взаимодействовать.
- Кнопка `Возродиться` отправляет `respawn_requested`; сервер переносит игрока к кресту respawn-а и восстанавливает `50%` от max HP.
- После смерти кабана исходная entity скрывается, становится non-solid и non-attackable до respawn-а.
- На месте смерти кабана появляется проходимый труп, который не attackable.
- Первый успешный `F` по трупу выдает `Кожа x2`, после чего loot у трупа отключается.
- Труп кабана исчезает через 5 минут server-time.
- Кабан respawn-ится через 1 минуту server-time с `50%` HP у своей домашней точки.
- Если игрок нажимает `H` без бинтов, сервер отправляет `Нет бинтов` в нижний лог и журнал.
- Если игрок нажимает `H` при полном HP, сервер отправляет `Вы не ранены` в нижний лог и журнал и не списывает бинт.
- При валидном применении бинта сервер сразу списывает `Бинт x1`, сбрасывает текущий player attack swing, но сохраняет выбранную auto-attack цель.
- Во время pending-перевязки движение, чат, `attack_requested` и `stop_attack_requested` разрешены; повторный бинт, interaction, interaction option, equip/unequip и vendor buy игнорируются до завершения.
- Входящий урон сейчас не сбивает перевязку; interrupt от урона с зависимостью от Healing записан в TODO.
- Длительность перевязки от Healing: `2.0s` до `15.0%`, далее линейно уменьшается до `1.5s` при `40.0%`.
- Шанс успеха Healing: `0.3` до `15.0%`, далее линейно растет до `0.8` при `40.0%`.
- Нижняя граница лечения бинтом: `1 + floor(HealingPercent / 10)`.
- Верхняя граница лечения бинтом: `3 HP` до `10.0%`, далее кусочно-линейно через `10.0% -> 6 HP`, `20.0% -> 10 HP`, `30.0% -> 15 HP`; до demo-cap остается `15 HP`.
- При успешном бинте игрок видит feed/journal-сообщение вида `вылечил +3 бинтом`, а после завершения перевязки attack swing запускается заново с полным cooldown оружия.
